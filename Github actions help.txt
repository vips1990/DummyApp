## GitHub Actions Workflow Explanation

This is a **GitHub Actions workflow file** — it's like an automated script that runs on GitHub's servers whenever you push code or merge a pull request. Think of it as a robot that builds and deploys your app automatically, so you don't have to do it manually every time.

### Quick Background: What is GitHub Actions?
- **GitHub Actions** is a free tool built into GitHub that lets you automate tasks like running tests or deploying to servers.
- Workflows are defined in YAML files stored in your repo under `.github/workflows/`.
- They run on virtual machines (called "runners") provided by GitHub, triggered by events like pushes or pull requests.

---

### Breaking Down This Specific Workflow

The file is `ci-deploy.yml` and defines a pipeline with two jobs: **build** and **deploy**.

#### 1. **Trigger: When Does This Run?**
```yaml
on:
  push:
  pull_request:
    types: [opened, synchronize, reopened, closed]
```
- **Build** runs on every push to any branch and on every pull request event.
- **Deploy** only runs when a pull request is **merged into `main`** (controlled by an `if:` condition on the deploy job).
- This means: pushing to a feature branch runs CI (build/test) but does NOT deploy. Only merging to `main` triggers a deployment.

---

#### 2. **Job 1: `build` (Build & Test Your App)**
```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - Checkout code
      - Set up Python 3.12
      - Install dependencies (pip install -r requirements.txt)
      - Run tests (placeholder — no tests yet)
```
- Runs on a free Ubuntu virtual machine from GitHub.
- Ensures your code and dependencies install cleanly before any deployment happens.
- If this fails, the deploy job will not run.

---

#### 3. **Job 2: `deploy` (Deploy to Azure VM)**
```yaml
  deploy:
    needs: build
    if: github.event_name == 'pull_request' && github.event.pull_request.merged == true && github.event.pull_request.base.ref == 'main'
```
- Only runs after `build` succeeds AND a PR has been merged into `main`.
- Uses the `appleboy/ssh-action` to connect to your Azure VM over SSH and run a deploy script remotely.

**What the deploy script does on the VM:**
1. `cd /home/<user>/app` — go to the app folder
2. `git fetch origin main && git reset --hard origin/main` — pull latest code (hard reset ensures clean state)
3. `python3 -m pip install --user -r requirements.txt` — install/update dependencies
4. `pkill -f "gunicorn.*app:app" || true` — stop any running gunicorn process
5. `nohup setsid python3 -m gunicorn --bind 127.0.0.1:8000 app:app &` — restart gunicorn bound to localhost only

**Why localhost only?** Gunicorn binds to `127.0.0.1:8000` (not publicly accessible). Nginx (set up by Ansible) acts as a reverse proxy, receiving traffic on port 80 and forwarding it to gunicorn internally. This is more secure than exposing gunicorn directly.

---

### How It All Works Together

```
Developer pushes to feature branch
        ↓
  Build job runs (CI check) — no deploy

Developer opens PR → merges into main
        ↓
  Build job runs first
        ↓ (if build passes)
  Deploy job SSHs into Azure VM
        ↓
  Code updated, gunicorn restarted
        ↓
  Nginx serves app at http://<VM-IP>:80
```

---

### Infrastructure: What Ansible Sets Up (One-Time)

Before GitHub Actions can deploy, the Azure VM must be provisioned by running the Ansible playbook (`ansible-create-vm.yml`). Ansible handles all infrastructure setup:
- Creates Azure resource group, VNet, subnet, NSG, public IP, NIC, and VM
- Installs: git, python3, python3-pip, python3-venv, gunicorn, nginx
- Configures Nginx as a reverse proxy (port 80 → localhost:8000)
- Clones the app repo into `/home/azureuser/app`

GitHub Actions only handles **code updates** after initial setup — it does not touch infrastructure.

---

### GitHub Secrets Required
Add these in repo Settings → Secrets and variables → Actions:
- `AZURE_VM_HOST`: Your VM's public IP (e.g., `123.45.67.89`)
- `AZURE_VM_USER`: SSH username (e.g., `azureuser`)
- `AZURE_VM_SSH_KEY`: Private SSH key content
- `AZURE_VM_PORT`: Usually `22`

---

### Why This Approach?
- **Automation**: Merge a PR and the app deploys itself.
- **Safety**: Only merged PRs to `main` trigger deploys — not every push.
- **Security**: Gunicorn is not exposed directly; Nginx acts as the public-facing layer. Secrets are stored in GitHub, never in code.
- **Clear separation**: Ansible owns infrastructure setup; GitHub Actions owns code deployment.
