## GitHub Actions Workflow Explanation

This is a **GitHub Actions workflow file** — it's like an automated script that runs on GitHub's servers whenever you push code to your repository. Think of it as a robot that builds, tests, and deploys your app automatically, so you don't have to do it manually every time.

### Quick Background: What is GitHub Actions?
- **GitHub Actions** is a free tool built into GitHub that lets you automate tasks like running tests, building apps, or deploying to servers.
- Workflows are defined in YAML files (like this one) stored in your repo under `.github/workflows/`.
- They run on virtual machines (called "runners") provided by GitHub, triggered by events like code pushes or pull requests.

### Breaking Down This Specific Workflow

The file is named something like `ci-deploy.yml` and defines a pipeline with two main parts: **building** your app and **deploying** it to an Azure VM.

#### 1. **Trigger: When Does This Run?**
```yaml
on:
  push:
    branches:
      - main
```
- This workflow **triggers automatically** whenever you push code to the `main` branch.
- No manual start needed — GitHub detects the push and starts the workflow.

#### 2. **Jobs: The Two Main Tasks**
Jobs are like steps in a recipe. This workflow has two jobs that run in sequence.

##### **Job 1: `build` (Build & Test Your App)**
```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.12'

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: Run tests (if any)
        run: |
          echo "No tests configured yet"
```
- **Runs on**: A free Ubuntu (Linux) virtual machine from GitHub.
- **What it does**:
  - **Checkout**: Downloads your repository code to the virtual machine.
  - **Set up Python**: Installs Python 3.12 on the machine.
  - **Install dependencies**: Upgrades pip (Python's package manager) and installs packages from `requirements.txt` (like Flask, etc.).
  - **Run tests**: Currently just prints a message (no tests are set up yet). In a real app, this would run unit tests to catch bugs early.
- **Purpose**: Ensures your code builds correctly and dependencies install without errors before deploying.

##### **Job 2: `deploy` (Deploy to Azure VM)**
```yaml
  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Deploy to Azure VM via SSH
        uses: appleboy/ssh-action@v0.1.10
        with:
          host: ${{ secrets.AZURE_VM_HOST }}
          username: ${{ secrets.AZURE_VM_USER }}
          key: ${{ secrets.AZURE_VM_SSH_KEY }}
          port: ${{ secrets.AZURE_VM_PORT }}
          script: |
            set -e
            cd /home/${{ secrets.AZURE_VM_USER }}/app
            git pull origin main
            python -m pip install --upgrade pip
            pip install -r requirements.txt
            # Run the app with gunicorn (adjust bind/worker count as needed)
            pkill -f "gunicorn.*app:app" || true
            nohup gunicorn --bind 0.0.0.0:8000 app:app &
```
- **Runs on**: Another Ubuntu virtual machine.
- **Dependency**: `needs: build` means it only runs **after** the build job succeeds. If build fails, deploy won't happen.
- **What it does**:
  - **Checkout**: Downloads your code again (needed for this job).
  - **Deploy via SSH**: Uses a pre-built action to connect to your Azure VM securely.
    - **Secrets**: Pulls sensitive info (VM IP, username, SSH key, port) from GitHub's encrypted secrets (you set these in repo settings — never hardcode them!).
    - **Script on VM**: Runs these commands remotely on your Azure VM:
      - `cd /home/${{ secrets.AZURE_VM_USER }}/app`: Go to the app folder on the VM.
      - `git pull origin main`: Pull the latest code from GitHub.
      - Install/update dependencies.
      - Kill any existing app process (`pkill`).
      - Start the app with **Gunicorn** (a production web server for Python apps) on port 8000, running in the background (`nohup`).
- **Purpose**: Automatically updates and restarts your app on the Azure VM after every push to `main`.

### How It All Works Together
1. You push code to `main` → Workflow starts.
2. **Build job** runs first: Sets up environment, installs stuff, runs tests.
3. If build passes, **deploy job** runs: SSH into Azure VM, pull code, install deps, restart app.
4. Your app is now live on the VM at `http://<VM-IP>:8000`.

### Setup Requirements
- **GitHub Secrets**: Add these in your repo settings (Settings → Secrets and variables → Actions):
  - `AZURE_VM_HOST`: Your VM's public IP (e.g., `123.45.67.89`).
  - `AZURE_VM_USER`: SSH username (e.g., `azureuser`).
  - `AZURE_VM_SSH_KEY`: Private SSH key content (generate with `ssh-keygen`).
  - `AZURE_VM_PORT`: Usually `22`.
- **VM Setup**: Your Azure VM needs Git, Python 3.12+, and the app cloned in `/home/<user>/app`.

### Why This is Useful
- **Automation**: No more manual deploys — just push code and it's live.
- **Reliability**: Tests catch issues before deployment.
- **Security**: Secrets keep credentials safe.
- **Free**: GitHub provides free minutes for public repos.
